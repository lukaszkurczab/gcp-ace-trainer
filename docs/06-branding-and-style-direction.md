# 06 — Branding and Style Direction

## Status dokumentu

Ten dokument definiuje kierunek marki, tonu komunikacji i stylu wizualnego dla wielotrackowej aplikacji treningowej.

Nie opisuje brandingu pojedynczego tracka egzaminacyjnego. GCP ACE jest jednym z tracków produktowych, a nie nazwą całej aplikacji.

Branding powinien wspierać model:

```txt
app brand → track identity → session mode → training item → feedback → progress
```

## Cel brandingu

Branding ma pozycjonować aplikację jako spokojne, techniczne narzędzie do systematycznego treningu umiejętności i przygotowania do zadań wymagających precyzyjnego myślenia.

Aplikacja powinna pasować zarówno do:

- przygotowania certyfikacyjnego, np. Google Cloud Associate Cloud Engineer,
- treningu rozwiązywania problemów algorytmicznych / LeetCode-style,
- późniejszych tracków technicznych, jeśli zostaną dodane.

Nie chodzi o budowę dużej marki konsumenckiej ani platformy edukacyjnej typu kursowego. Na tym etapie branding ma zapewnić spójność UI, komunikacji, nazewnictwa i decyzji stylistycznych.

## Zasada nadrzędna

Marka aplikacji powinna być neutralna domenowo.

Nie wolno wiązać nazwy, tonu ani głównej obietnicy wyłącznie z:

- GCP,
- egzaminem,
- chmurą,
- pytaniami testowymi,
- LeetCode,
- kodowaniem,
- jedną metodą nauki.

Track może mieć własną etykietę i kontekst, ale aplikacja jako całość powinna komunikować szerszą wartość: techniczny trening, powtarzanie, rozpoznawanie wzorców, feedback i postęp.

## Nazwa aplikacji

Docelowa nazwa produktu: **Patternly**.

Patternly jest nazwą parasolową dla całej aplikacji. Nazwa wynika z głównej wartości produktu: użytkownik ćwiczy techniczne myślenie przez rozpoznawanie wzorców, analizę błędów, powtarzalną praktykę i diagnozę gotowości.

Nazwa nie powinna być traktowana jako nazwa pojedynczego tracka. `GCP ACE`, `Cloud Certification`, `Algorithms` i `LeetCode-style` są kontekstami treningowymi, nie markami aplikacji.

## Relacja: marka aplikacji vs track

Aplikacja powinna mieć jedną nazwę parasolową.

Tracki mogą mieć opisowe nazwy:

```txt
Cloud Certification
Algorithms
```

lub bardziej szczegółowe etykiety:

```txt
GCP ACE Preparation
Algorithm Patterns
```

Tracki nie powinny wyglądać jak osobne aplikacje. Powinny korzystać z tego samego systemu UI, podobnej struktury sesji i wspólnych zasad feedbacku.

## Disclaimer i legal-brand safety

Dla tracków opartych o certyfikacje trzeba jasno komunikować brak afiliacji.

Przykład neutralny:

```txt
This app is an independent study tool and is not affiliated with or endorsed by Google.
```

Wersja polska:

```txt
Aplikacja jest niezależnym narzędziem do nauki i nie jest powiązana z Google ani przez Google zatwierdzona.
```

Dla tracka GCP można doprecyzować:

```txt
Google Cloud and related certification names are trademarks of Google LLC. This app is not affiliated with or endorsed by Google.
```

W UI nie należy sugerować, że aplikacja zawiera oficjalne pytania egzaminacyjne, gwarantuje zdanie egzaminu albo zastępuje oficjalną dokumentację.

Dla tracka algorytmicznego nie należy sugerować afiliacji z LeetCode ani kopiować nazw, treści lub układu platformy LeetCode.

## Osobowość marki

Marka powinna być:

- spokojna,
- techniczna,
- rzeczowa,
- precyzyjna,
- neutralna domenowo,
- pomocna,
- skupiona na feedbacku,
- odporna na hype.

Nie powinna być:

- gamingowa,
- krzykliwa,
- infantylna,
- zbyt korporacyjna,
- podobna do oficjalnych produktów Google,
- podobna do LeetCode,
- oparta o pseudo-motywacyjne komunikaty,
- oparta o presję i zawstydzanie użytkownika.

## Kierunek stylistyczny

Rekomendowany styl: **Focus Lab**.

Focus Lab oznacza ciemny, techniczny i skupiony interfejs, który wygląda jak osobiste laboratorium treningowe, nie jak kurs, quizownik, gra ani pełne IDE.

Podstawowe cechy:

```txt
dark-first interface
premium technical tone
restrained neon accents
clear hierarchy
one primary action per screen
low information density
large breathing space
focused session layouts
progressive disclosure
strong review states
```

W praktyce UI powinno stosować:

- ciemną bazę tła,
- subtelne gradienty i glow tylko jako akcent,
- mocne CTA, ale bez agresywnego koloru na każdym elemencie,
- małą liczbę kart na pierwszym widoku,
- krótkie etykiety zamiast opisowych bloków tekstu,
- duże odstępy między sekcjami,
- komponenty o wysokiej czytelności,
- progres i statusy jako pomocnicze wskaźniki, nie dominujące dekoracje.

Styl powinien być bliżej narzędzia pracy niż aplikacji edukacyjnej dla masowego odbiorcy.

Nie robić osobnych theme’ów dla każdego tracka. Track może mieć akcent, badge i ikonę, ale baza UI pozostaje wspólna.

### Redukcja gęstości informacji

Warianty stylistyczne wygenerowane na etapie eksploracji pokazały problem: ekrany łatwo stają się przeładowane kartami, wykresami, metrykami i tekstem. Dla Patternly jest to szczególnie szkodliwe, bo aplikacja ma wspierać skupienie.

Zasada projektowa:

```txt
One screen. One dominant job. One primary action.
```

Reguły praktyczne:

- Home pokazuje rekomendowaną kolejną akcję, skrót gotowości i kontynuację ostatniego kontekstu.
- Track selection pokazuje wybór ścieżki, nie dashboard całego produktu.
- Session screen skupia się na jednym training itemie.
- Feedback screen pokazuje wynik, przyczynę i następną akcję.
- Progress pokazuje diagnozę i słabe obszary, nie pełne centrum analityczne.
- Library jest strukturą pomocniczą, nie głównym ekranem wejścia.

Treści drugorzędne powinny trafiać do szczegółów, zakładek, bottom sheetów lub ekranów drill-down.

## Track accents

Akcenty tracków powinny być drugorzędne.

Przykład:

```txt
Cloud Certification → blue / cyan accent
Algorithms → violet / indigo accent
Review → amber accent
Correct / strong → green accent
Incorrect / weak → red accent
Primary CTA → violet gradient, used sparingly
```

Akcent nie może dominować nad czytelnością. Nie powinien też tworzyć wrażenia, że użytkownik przełącza się między różnymi produktami.

## Inspiracje jakościowe

Do traktowania jako kierunek jakości, nie jako wzór do kopiowania:

- Linear: klarowność, rytm UI, oszczędność komunikatów,
- Notion: spokojna organizacja treści,
- Raycast: szybka akcja i techniczny minimalizm,
- Vercel: prostota i premium technical tone,
- GitHub: techniczna czytelność i zaufanie,
- Brilliant: nauka przez problem, ale bez kopiowania stylu lub gamifikacji.

## Zasady copy

### Krótko

UI nie powinien używać długich tekstów tam, gdzie wystarczy etykieta.

### Konkretnie

Komunikat powinien mówić, co użytkownik ma zrobić albo czego się dowiedział.

### Domenowo, ale nie hermetycznie

Track może używać słownictwa domenowego, ale aplikacja jako całość nie powinna być zrozumiała tylko dla jednego typu użytkownika.

Przykład:

```txt
IAM needs review.
```

jest dobre wewnątrz tracka GCP, ale nie jako globalny komunikat Home bez kontekstu.

### Bez sztucznej ekscytacji

Motywacja powinna wynikać z jasności postępu, nie z krzykliwych komunikatów.

### Błąd jako informacja

Nie mówić użytkownikowi, że „failował”. Pokazać, co wymaga review i dlaczego.

### Feedback jako narzędzie uczenia

Feedback nie powinien ograniczać się do `Correct` / `Incorrect`.

W szczególności w Algorithms track feedback powinien wyjaśniać:

- rozpoznany lub pominięty pattern,
- błędną strategię,
- niepoprawną analizę złożoności,
- pominięte ograniczenie,
- trade-off między rozwiązaniami.

## Antywzorce brandingowe

Nie robić:

- nazwy aplikacji zależnej od GCP,
- nazwy aplikacji zależnej od LeetCode,
- głównej obietnicy opartej o zdanie egzaminu,
- głównej obietnicy opartej o przejście rozmowy rekrutacyjnej,
- UI stylizowanego na Google Cloud Console,
- UI stylizowanego na LeetCode,
- osobnych mini-brandów dla każdego tracka,
- agresywnej gamifikacji,
- przeładowanych ekranów z wieloma równorzędnymi kartami,
- komunikatów presji typu `you failed`,
- zbyt szerokiego claimu typu `master any technical topic`.

## Decyzje bazowe

- Nazwa aplikacji to Patternly.
- Marka aplikacji jest neutralna względem tracków.
- GCP ACE nie jest nazwą aplikacji.
- LeetCode-style / Algorithms nie jest osobną aplikacją.
- Tracki mogą mieć własne akcenty, ale nie własne pełne brandy.
- Kierunek stylistyczny to Focus Lab.
- UI jest dark-first, premium, skupiony i odchudzony informacyjnie.
- Ton jest techniczny, spokojny i konkretny.
- Feedback jest ważniejszy niż motywacyjne komunikaty.
- Aplikacja nie udaje oficjalnego symulatora egzaminu ani zamiennika platform coding interview.
